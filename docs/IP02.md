# Improvement Proposal 02: Closing the Prediction Gap

**Date:** 2026-03-03
**Status:** Proposed
**Precedes:** model_testing_3

---

## 0. Current State & Problem Statement

The v2 model (Random Forest, calibrated, 40 features, trained on 2022–2024) achieves:

| Metric | CV (2022–2024) | 2025 Holdout |
|--------|---------------|--------------|
| ROC-AUC | 0.782 | 0.785 |
| PR-AUC | 0.149 | 0.200 |
| Brier | 0.029 | 0.035 |

At the learned threshold (0.534), the model predicts **3 overtakes out of 551 actual** in the 2025 season — 0.5% recall. Even at a relaxed threshold of 0.10, recall is only 36.7% at 20% precision.

The core issue is **weak class separation**: mean predicted probability for true overtakes is 0.107, compared to 0.037 for non-overtakes — a separation of just 0.07. The model can rank overtakes above non-overtakes (decent ROC-AUC), but it cannot push the absolute probability high enough to cleanly separate them.

This proposal addresses the root causes across three axes: data quality, feature engineering, and model architecture.

---

## 1. Data Quality & Label Issues

### 1.1 `gap_ahead` semantics are inverted for 54% of rows (CRITICAL)

**Issue**
`gap_ahead` is computed as `attacker_lap_time - defender_lap_time` (line 127 of `battle_detector.py`). This measures the *pace difference on that lap*, not the actual *time gap on track*. Crucially:

- 54% of rows have `gap_ahead > 0` (attacker was slower on that lap) — meaning the attacker is falling behind, not attacking. These are not real attack situations.
- The overtake rate when `gap_ahead > 0` is only 1.9%, vs 6.3% when `gap_ahead < 0`.
- 146 "overtakes" happened with `gap_ahead > 0`, which may be position changes due to pit stops, not genuine on-track passes.

**Impact**
The model trains on 54% noise — rows where the attacker isn't actually closing in. This severely dilutes the overtake signal.

**Recommendation**
- Replace `gap_ahead` with the actual **inter-car time gap** from FastF1's `LapStartTime` difference or the gap information in `_extended_timing_data`. This is the real gap in seconds between the cars on track.
- As a fallback, use `attacker_info['gap']` already available from `build_position_and_gap_map`.
- Add a separate feature `pace_delta = defender_lap_time - attacker_lap_time` (positive = attacker is faster per lap).

---

### 1.2 Pit-stop overtakes pollute the label (HIGH)

**Issue**
An "overtake" is defined as the attacker having a lower position number on the next lap. This captures pit-stop position changes (undercuts, overcuts) as "overtakes". These are not genuine on-track passes — they are strategic moves that happen in the pit lane.

**Impact**
The model learns a mixture of two fundamentally different phenomena — on-track passing and pit-strategy shuffles — with the same label. This confuses the model and makes the "overtake" class internally inconsistent.

**Recommendation**
- Filter out battles where either driver is on a pit-in or pit-out lap (check `PitInTime` / `PitOutTime` fields, which are already available from FastF1).
- Alternatively, add a boolean feature `pit_stop_involved` so the model can distinguish the two types.
- Consider a more robust overtake definition: position improved AND the cars were within DRS range (< 1s actual gap) AND neither pitted.

---

### 1.3 Battle pair redundancy — 91% of rows are continuations (HIGH)

**Issue**
90.7% of rows are a continuation of the same attacker–defender pair from the previous lap. A pair persists for a median of 6 laps. Each lap is treated as an independent observation, but:

- The features barely change between consecutive laps (tyre age +1, small gap variation).
- The model sees the "same battle" 6 times but the overtake only happens once (or not at all).
- This massively inflates the negative class and makes the problem appear harder than it actually is.

**Impact**
- Extreme class imbalance (4% positive) is partly artificial — at the *battle-pair* level, 35.5% end in an overtake at some point.
- The model is penalised for predicting "overtake" on the laps where the attacker is building up to the pass — these are technically correct warnings, but labelled negative.

**Recommendation (two complementary approaches):**

**A. Aggregated prediction (recommended):**
Restructure the problem: predict whether a **battle pair** (attacker A vs defender B for N consecutive laps) will result in an overtake. Each pair becomes one row with aggregated features:
- Min/mean/last gap, pace delta trend (slope), tyre age at start/end
- Whether gap is closing (binary), how many laps the battle persisted
- This converts the problem from 14,071 rows at 4% positive to ~1,314 rows at 35.5% positive — a far more balanced and learnable problem.

**B. Sequence-aware windowed features (alternative):**
Keep per-lap granularity but add rolling/lagged features from the previous 2–3 laps:
- `gap_change_1lap`, `gap_change_3lap` (is the gap closing?)
- `pace_delta_rolling_3` (who was faster recently?)
- `attack_intensity` = laps within DRS range out of last 3
- This gives the model temporal context it currently completely lacks.

---

### 1.4 `tyre_age_difference` uses absolute value — drops directionality

**Issue**
Line 142: `tyre_age_difference = abs(attacker_tyre_age - defender_tyre_age)`. Using `abs()` loses the crucial information of *who* has the tyre advantage.

**Recommendation**
Use signed difference: `attacker_tyre_age - defender_tyre_age`. Negative = attacker on fresher tyres = more likely to overtake.

## 2. Feature Engineering

### 2.1 Gap trend / closing rate (HIGH)

The single most informative pre-overtake signal, found in diagnostics: the gap between cars narrows in the laps before an overtake. Currently no feature captures this.

**Proposed features:**
```
gap_delta_1  = gap[lap] - gap[lap-1]        # gap change in last lap
gap_delta_3  = gap[lap] - gap[lap-3]        # gap change over 3 laps
is_closing   = gap_delta_1 < 0              # binary: is the attacker gaining?
closing_laps = count of last N laps where gap decreased  # momentum
```

---

### 2.2 Pace advantage features (HIGH)

The current `gap_ahead` conflates gap and pace. These should be separate:

```
actual_gap        = inter-car time gap (seconds) from timing data
pace_delta        = defender_lap_time - attacker_lap_time (positive = attacker faster)
pace_delta_avg_3  = rolling 3-lap average pace delta
```

---

### 2.3 Relative speed (not absolute)

The model currently gets 8 absolute speed-trap features (4 per driver). The model's top features are `gap_ahead` and `attacker_straight_speed` — but the relative differences are more meaningful.

**Proposed features:**
```
speed_i1_delta = attacker_speed_i1 - defender_speed_i1
speed_i2_delta = attacker_speed_i2 - defender_speed_i2
speed_fl_delta = attacker_finish_line_speed - defender_finish_line_speed
speed_st_delta = attacker_straight_speed - defender_straight_speed
```

Keep 1–2 absolute speed features (e.g. max straight speed) for track context, but the deltas should be primary.

---

### 2.4 Battle context features

```
battle_duration     = laps this attacker-defender pair has been battling
attempted_before    = has this pair had an overtake earlier in the race? (boolean)
overtakes_this_race = total overtakes already happened in this race so far
position_changes_last_5_laps = count of all position changes recently (measures chaos)
```

---

### 2.5 Tyre cliff & compound advantage

```
compound_advantage  = expected_pace[attacker_compound] - expected_pace[defender_compound]
tyre_cliff_risk     = 1 if defender_tyre_age > cliff_threshold[compound] else 0
attacker_on_newer_stint = attacker_stint > defender_stint  # just pitted = strategic advantage
```

Approximate tyre cliff thresholds (from F1 domain knowledge): SOFT ~18 laps, MEDIUM ~28 laps, HARD ~38 laps.

---

### 2.6 Driver skill proxy (without identity leakage)

Driver identity was removed in IP01, but some skill signal is valid:

```
attacker_season_overtake_rate = driver's overtake success rate so far this season
defender_season_defend_rate   = driver's successful defence rate so far
attacker_championship_pos     = current championship standing
```

These use aggregate performance metrics, not the driver's name, so they generalise to future seasons and new drivers.

---

### 2.7 Telemetry micro-features (if available)

FastF1 provides car telemetry at ~4Hz. For battles that enter a DRS detection zone:

```
drs_activated       = was DRS actually opened? (boolean)
speed_at_braking    = attacker's speed at braking point before the corner
min_sector_speed    = minimum speed through the preceding corner complex
```

These are computationally expensive to extract but directly capture "is this car set up for a pass on this specific lap?"

## 3. Model Architecture

### 3.1 Gradient Boosting with proper tuning (HIGH)

The current Random Forest uses default hyperparameters. The v2 Gradient Boosting actually had a lower Brier score (0.044 vs 0.110) despite lower ROC-AUC, suggesting it's better calibrated.

**Recommendation:**
- Use XGBoost or LightGBM with Bayesian hyperparameter tuning (Optuna).
- Key parameters to tune: `max_depth`, `learning_rate`, `n_estimators`, `min_child_weight`, `subsample`, `colsample_bytree`, `scale_pos_weight`.
- `scale_pos_weight = n_negative / n_positive ≈ 23` for direct class imbalance handling.

---

### 3.2 Sequence model for battle trajectories (MEDIUM)

Overtakes are inherently sequential events — the outcome depends on how the battle *evolved* over the preceding laps, not just the current snapshot. A sequence model can capture:

- Gap closing patterns (the "catch phase" → "attack phase" → "pass or fail" arc)
- DRS activation patterns across multiple laps
- Tyre degradation trajectories

**Options (increasing complexity):**

| Model | Complexity | Expected Benefit |
|-------|-----------|-----------------|
| LSTM / GRU on 3–5 lap windows | Medium | Captures closing dynamics |
| 1D-CNN on lap sequence | Medium | Fast pattern extraction |
| Transformer encoder on full battle | High | Attention over variable-length battles |

For the aggregated battle-pair approach (§1.3A), a simple MLP on the aggregate features may suffice.

---

### 3.3 Two-stage model (MEDIUM)

The prediction task is really two questions:
1. **Will an attack happen?** (Is the attacker in a position to attempt a pass?)
2. **Will the attack succeed?** (Given an attempt, does it result in a position change?)

Currently these are conflated into one binary label. A two-stage approach:

- **Stage 1 — Attack detector:** classify whether this lap represents an "attack attempt" (features: gap < 1s, pace delta, DRS available, closing rate).
- **Stage 2 — Success predictor:** among attacks, predict success (features: speed delta in DRS zone, tyre advantage, track type).

This decomposition makes each sub-problem much more learnable.

---

### 3.4 Threshold optimisation for practical use (HIGH)

The current threshold (0.534) maximises F1-score in cross-validation, but the use case is probabilistic — we want the model to *flag likely overtakes*, not maximise a binary classification metric.

**Recommendation:**
- Optimise the threshold for a practical operating point, e.g.:
  - Target recall ≥ 50% (catch at least half the overtakes)
  - Minimise false positives subject to that recall target
- From diagnostics: threshold = 0.10 gives 37% recall at 20% precision — still useful for flagging.
- Report results at multiple operating points, not just one threshold.
- Consider a **cost-sensitive threshold**: in a broadcast/betting context, missing a real overtake (false negative) is more costly than a false alarm.

---

### 3.5 Ensemble / stacking (LOW)

Combine the strengths of different model types:

```
meta_model = LogisticRegression()
base_models = [
    GradientBoosting(focus: gap/speed features),
    RandomForest(focus: categorical/track features),
    LSTM(focus: temporal sequence),
]
```

Stacking typically improves PR-AUC by 0.02–0.05 on imbalanced problems.

## 4. Evaluation & Calibration

### 4.1 Evaluate at the battle-pair level (HIGH)

Even if the model predicts per-lap, the practical question is: "did this battle end in an overtake?" Evaluate by:
1. For each battle pair, take the **max predicted probability** across laps.
2. Evaluate ROC-AUC and PR-AUC at the pair level.
3. This converts the problem from 14k rows at 4% to ~1.3k pairs at 35.5%, and metrics will better reflect real-world utility.

---

### 4.2 Temporal calibration check

Calibration should be validated per-year, not just overall. The model may be well-calibrated on 2022 data but miscalibrated on 2024 due to regulation changes.

---

### 4.3 Track-stratified evaluation

Report metrics per track type (high-speed, street, technical) since overtaking dynamics differ dramatically. Monaco should not be in the same bucket as Monza.

---

## 5. Data Collection Improvements

### 5.1 Add 2022–2025 Sprint races

Sprint races have different dynamics (shorter, no pit stops) but provide additional overtake data. Tag them with `session_type=sprint` so the model can learn the difference.

---

### 5.2 Use timing micro-sector data

FastF1 provides timing data at sector level. A driver might be faster in sector 1 but slower in sector 3. Sector-level pace comparison is more informative than whole-lap time comparison for predicting where an overtake will happen.

---

### 5.3 Incorporate race control messages

FastF1's `race_control_messages` include:
- Track limits warnings (driver under pressure makes mistakes)
- DRS enabled/disabled messages (exact lap when DRS becomes available)
- Blue flag notifications (lapped traffic affecting battles)

## 7. Realistic Target Metrics

Given the inherent unpredictability of F1 overtakes, the theoretical ceiling is well below 100% accuracy. Reasonable targets after implementing P0–P2:

| Metric | Current | Target (per-lap) | Target (per-pair) |
|--------|---------|------------------|--------------------|
| ROC-AUC | 0.785 | 0.82–0.85 | 0.85–0.90 |
| PR-AUC | 0.200 | 0.30–0.35 | 0.55–0.65 |
| Recall @ 20% precision | 15% | 40–50% | 65–75% |
| Brier | 0.035 | 0.030 | 0.22 |

The per-pair framing is the more natural and practically useful evaluation.
