# Improvement Proposal 01: Overtake Probability Model Pipeline

This document enumerates detected issues and proposes concrete improvements.

---

## 1. Syntax & Implementation Issues

### 1.1 CSV naming and season ambiguity

**Issue**
- The input file name suggests `2025`, while the intended training range is `2022–2024`.
- This creates ambiguity about whether future races are included.

**Impact**
- Potential temporal leakage.
- Invalid offline evaluation.

**Recommendation**
- Enforce explicit season filtering before training.
- Encode season range in the filename and/or dataset metadata.

Example:
```
battles_2022_2024.csv
```

---

### 1.2 Redundant numeric features

**Issue**
The dataset includes:
- `attacker_speed`
- `defender_speed`
- `speed_difference`

**Impact**
- Strong collinearity.
- Unstable coefficients for linear models.
- Poor probability calibration.

**Recommendation**
- Prefer difference-based features only.
- Drop absolute speeds unless strictly required.

---

## 2. Semantic / ML-Level Bugs

### 2.1 Driver identity leakage (CRITICAL)

**Issue**
Categorical features include:
- `attacker`
- `defender`

**Impact**
- The model learns *who* overtakes, not *why* overtakes happen.
- Predictions collapse for rookies, driver changes, and future seasons.
- Inflated offline metrics.

**Recommendation**
- Remove driver identifiers entirely, or
- Replace with derived attributes:
  - driver Elo / rating
  - season championship position
  - recent overtakes per race

---

### 2.2 High-cardinality one-hot encoding

**Issue**
OneHotEncoding is applied to:
- drivers
- tracks
- tyre compounds

**Impact**
- Extremely sparse feature matrix.
- Overfitting.
- Poor generalization.
- Unstable probability estimates.

**Recommendation**
- Reduce cardinality:
  - drop driver IDs
  - encode track metadata numerically
- Prefer tree-based models if categorical encoding is unavoidable.

---

### 2.3 Probability misinterpretation

**Issue**
Models are used directly for probability output without calibration.

**Impact**
- Output values are not true probabilities.
- A predicted 0.8 does not correspond to an 80% chance of overtake.

**Recommendation**
- Apply explicit probability calibration:
```
CalibratedClassifierCV(model, method="isotonic")
```
- Validate using reliability / calibration curves.

---

## 3. Data Splitting & Evaluation Issues

### 3.1 Temporal leakage (CRITICAL)

**Issue**
Random train-test splitting is applied to time-ordered race events.

**Impact**
- Battles from the same race/session appear in both train and test sets.
- The model indirectly sees future context during training.
- Metrics are overly optimistic.

**Recommendation**
- Use season-based splits:
  - Train: 2022–2023
  - Test: 2024
- Or GroupKFold by race or session ID.

---

### 3.2 Inappropriate primary metrics

**Issue**
Accuracy is used as a primary metric.

**Impact**
- Overtakes are rare events.
- A trivial "no overtake" model achieves misleadingly high accuracy.

**Recommendation**
Prioritize:
- ROC-AUC
- Precision-Recall AUC
- Calibration curves
- Brier score

Accuracy should be secondary or removed.

---

## 4. Big-Picture Modeling Issues

### 4.1 Mixing descriptive and predictive modeling

**Issue**
Including driver names turns the model into a descriptive model of historical dominance rather than a predictive model of race dynamics.

**Impact**
- Results are not suitable for future prediction or simulation.

**Recommendation**
Explicitly choose:
- Predictive model → remove identities, focus on physics & context.
- Descriptive model → keep identities, but label accordingly.

---

### 4.2 Track information under-utilization

**Issue**
Tracks are treated as categorical labels rather than structured metadata.

**Impact**
- Poor generalization across circuits.
- Loss of domain knowledge.

**Recommendation**
Replace raw track names with:
- track_type
- sector_type
- drs_zone_length
- number_of_drs_zones

---

### 4.3 No class imbalance handling

**Issue**
Overtakes are rare events, but no imbalance handling is applied.

**Impact**
- Biased decision boundary.
- Poor recall for overtakes.

**Recommendation**
- Use class_weight="balanced" where supported.
- Tune decision thresholds based on use case.

---
