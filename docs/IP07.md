# Improvement Proposal 07: Broad Scenario-Led Overtake Prediction with v6 Data

**Date:** 2026-04-01  
**Status:** Implemented (`pipeline/v6_pipeline.py`, `data/v6`, `notebooks/model_testing_6.ipynb`)  
**Builds on:** IP03, IP04, IP06, and the wide-coverage audit in `notebooks/pipeline_testing.ipynb`

---

## 0. Why v6 exists

v1-v5 framed the task as:

- adjacent pair only,
- gap under 1.0 s,
- predict whether the attacker passes on the next lap.

That framing was practical, but too narrow:

1. It misses many real overtakes that begin before the pair satisfies the strict battle threshold.
2. It penalizes the model for being "too early" when an attack takes 2-3 laps to complete.
3. It undercounts overall overtake activity compared with published season references.

To test the size of that undercount, `notebooks/pipeline_testing.ipynb` switched to a much wider extraction rule based on **pairwise order flips between consecutive laps**. With notebook-selected filters (`exclude_pit_related=True`, everything else left permissive), the recovered season totals were close to public reference totals while still remaining conservative enough for modelling.

That notebook gave the key design decision for v6:

- keep the problem **supervised**,
- use the notebook logic to define **positive overtake events**,
- generate a broader set of **candidate scenarios** that may or may not turn into those events.

---

## 1. New problem framing

v6 is still a **binary classification** problem, but the unit of prediction changes.

### Old framing (v1-v5)

> "Given a very narrow battle row, will the attacker overtake on the next lap?"

### New framing (v6)

> "Given a broader adjacent attacker-defender scenario at lap `L`, will a filtered on-track overtake occur within the next 3 laps?"

This keeps the task predictive, but aligns it much better with how overtakes build in real races.

---

## 2. Positive labels: notebook-style overtake extraction

`pipeline/v6_pipeline.py` ports the wide extractor from `notebooks/pipeline_testing.ipynb` into reusable pipeline code:

- load races by **`EventName`**,
- include all race weekends with **`RoundNumber > 0`**,
- compare every shared driver pair between lap `L` and `L+1`,
- mark a raw event when their relative order flips,
- attach filter metadata:
  - `pit_related`
  - `accurate_timing`
  - `safety_car`
  - `yellow_flag`
  - `lap1_or_restart_like`
  - adjacency before/after
  - `position_gain`

### Positive filter used in v6

The dataset uses the same permissive filter family that worked best in the notebook audit:

```text
exclude_pit_related = True
exclude_lap1 = False
exclude_safety_car = False
exclude_yellow_flag = False
require_accurate_timing = False
adjacency_rule = "none"
max_position_gain = None
```

This means v6 positives are:

- wide enough to capture more true overtakes than the old battle detector,
- filtered enough to drop obvious pit-stop artifacts.

---

## 3. Candidate scenarios

Positive events alone are not enough for supervised learning. v6 therefore creates a broader **candidate universe**:

- adjacent attacker-defender pair on lap `L`,
- both cars on track,
- same lap,
- attacker within **3.0 s** of the defender,
- race context and pair-state features available at lap `L`.

This is broader than v1-v5 because it does **not** require the old `<1.0 s` battle threshold and the primary target is no longer only "next lap".

### Labels written to v6 rows

- `overtake_next_lap`
- `overtake_within_2`
- `overtake_within_3`
- `label` = `overtake_within_3` (primary v6 target)

This keeps the dataset reusable while making the default modelling target more learnable.

---

## 4. Feature set

v6 keeps the strong v4/v5 signals and adds richer scenario dynamics.

### Kept from v4/v5

- lap-time, pace, speed-trap, tyre, stint, DRS, track, weather
- sector deltas and strongest sector
- team and teammate-relative form features
- constructor ranks from IP06
- race-situation context: `gap_to_car_ahead`, `gap_to_car_behind`, `drs_train_size`, `race_phase`, `stint_phase`

### New or emphasised in v6

- `label` based on `overtake_within_3`
- `gap_delta_1`, `gap_delta_2`, `gap_delta_3`
- `gap_mean_3`, `gap_min_3`
- `pace_delta_avg_3`, `pace_delta_std_3`
- `speed_st_delta_avg_3`
- `battle_duration`
- `prior_pair_overtakes`
- `overtakes_so_far`
- `gap_pressure_ratio`, `rear_pressure_ratio`
- `gap_to_leader`
- `laps_remaining`

These features push the model toward scenario dynamics instead of only static snapshots.

## 6. Training strategy

`notebooks/model_testing_6.ipynb` trains on:

- `2022`
- `2023`
- `2024`

and tests on:

- `2025`

The notebook uses:

- `data/v6/scenarios_*.csv`
- primary target `label` (`overtake_within_3`)
- pit-stop candidate filtering before training
- XGBoost on mixed numeric/categorical tabular features

This is intentionally still a tree-based tabular model, not a neural network first approach. The dataset is structured, the features are heterogeneous, and gradient-boosted trees remain the best baseline for this setting.

---

## 7. Important interpretation note

v6 metrics are **not directly apples-to-apples** with v1-v5 metrics.

Why:

- the candidate universe is broader,
- the label is softer (`within_3`, not just next lap),
- the positive set is based on notebook-style filtered overtakes rather than the old detector.

So v6 should be interpreted as a **new supervised problem definition**, not just another feature iteration on the old one.
