# Improvement Proposal 04: Context-Aware Driver Form and Productised Inference

**Date:** 2026-03-20
**Status:** Implemented (model & dataset: v5 / `data/v5` / `model_testing_5.ipynb`; product UI & API overhaul: see IP05)
**Precedes:** model_testing_5

## 0. Current State After IP03

The v4 model (XGBoost, SHAP-pruned, trained on `data/v4`, with pair-level evaluation and IP03 feature additions) improved modestly over v3 on the per-lap task:

| Metric | v3 (IP02) | v4 (IP03) | Delta |
|--------|-----------|-----------|-------|
| ROC-AUC (CV) | 0.865 | 0.869 | +0.004 |
| PR-AUC (CV) | 0.384 | 0.394 | +0.010 |
| Brier (CV) | 0.122 | 0.098 | -0.024 |
| ROC-AUC (2025) | 0.888 | 0.894 | +0.006 |
| PR-AUC (2025) | 0.467 | 0.469 | +0.002 |
| Brier (2025) | 0.037 | 0.036 | -0.001 |

**Battle-pair model (2025 holdout):** ROC-AUC ~0.887, PR-AUC ~0.710.

### What improved in v4

- `track_type` is no longer silently collapsed to `street`.
- Sector-level micro-features and driver rolling rates add richer race context.
- SHAP pruning reduced noise and improved calibration.
- Pair-level modelling remains the strongest framing for practical use.

### What still limits the model

1. **Car/package strength is under-modelled.** Formula 1 overtakes are heavily conditioned by the constructor package, straight-line efficiency, drag level, tyre warm-up behaviour, and upgrade cycle. The current model only sees indirect traces of this through lap times and speed traps.
2. **Driver form is only partially represented.** Rolling rates over the last five races help, but they do not capture *current-season trend*, quali-vs-race delta, or how a driver is performing relative to teammate and car baseline.
3. **Race situation is still too flat.** The model knows lap-level battle state, but not enough about the broader race phase: tyre offset rarity, pit-window pressure, DRS train dynamics, or whether the attacker is in a faster car trapped out of position.
4. **The app is still notebook-era architecture.** `app.py` mixes UI, feature engineering, model loading, and inference logic in one file, which makes it harder to test, extend, or serve through other interfaces.
5. **The UI communicates a score, not a decision context.** Users see a probability, but not why the model thinks the chance is high or low, what assumptions were engineered, or how the prediction changes under different scenarios.

This proposal targets the next stage: make the model more *situation-aware* and the application more *product-like*.

## 1. Next Model Features

### 1.1 Constructor / team performance features (HIGH)

The user observation is correct: **the car itself is a major feature** in overtaking likelihood. A Red Bull-style high-top-speed package, a Williams low-drag setup, or a Ferrari traction profile materially changes the quality of an attack even before driver skill is considered.

**Proposal:** add team/package context without leaking raw future outcomes.

Candidate features:

```text
attacker_team, defender_team
same_team_battle                        # teammate battle dynamics differ
team_power_rank                         # season-to-date straight-line speed rank
team_race_pace_rank                     # season-to-date long-run pace rank
team_quali_pace_rank                    # season-to-date one-lap pace rank
team_drag_index                         # finish-line / straight-line speed profile
team_upgrade_phase                      # early / mid / late season package proxy
team_relative_delta                     # attacker team minus defender team rank
```

**Leak-safe implementation idea:**
- Map each driver to constructor per season.
- Compute team-level aggregates using only prior races in that season.
- For the first few rounds, fall back to previous season end-state or neutral priors.

**Why this matters:** many overtakes are not just “driver A faster than driver B”; they are “car A is structurally better in this race situation.”

---

### 1.2 Within-season driver form, not just 5-race averages (HIGH)

The current rolling attacker/defender rates are useful, but still too coarse.

**Proposal:** add richer driver-season form signals:

```text
attacker_form_last_3_races
attacker_form_last_8_races
defender_form_last_3_races
driver_quali_minus_teammate
driver_race_finish_minus_teammate
driver_positions_gained_per_race
driver_overtake_success_this_season
driver_defence_success_this_season
driver_first-stint_attack_rate
driver_late-stint_attack_rate
```

A stronger version uses **teammate-relative** features:

```text
attacker_race_pace_vs_teammate
attacker_quali_vs_teammate
defender_race_pace_vs_teammate
defender_positions_gained_vs_teammate_baseline
```

These capture whether the driver is currently overperforming or underperforming the underlying car.

**Why this matters:** if a driver is consistently beating teammate in race pace, that is a stronger live signal than a raw identity feature and does not force the model to memorise names.

---

### 1.3 Race-situation features beyond the local battle window (HIGH)

Overtakes depend on the broader **situation**, not only the two-car local state.

**Proposal:** add event-level context:

```text
cars_in_drs_train                     # attacker and defender stuck in train?
gap_to_car_ahead_of_defender          # can defender attack instead of defend?
gap_to_car_behind_attacker            # attacker under pressure?
pit_window_open                       # race entering undercut/overcut phase
tyre_offset_rarity                    # unusual tyre mismatch this race
stint_phase                           # early / mid / cliff / extended
race_phase_bucket                     # opening / middle / final stint phase
safety_car_since_last_5_laps
yellow_flag_since_last_3_laps
track_position_value_proxy            # hard-to-pass circuit vs easy-to-pass
```

Also consider:
- whether the attacker has already failed multiple attacks this stint,
- whether the defender is pace-limited by damage / traffic proxy,
- whether both cars are in the same DRS train.

**Expected impact:** better separation between “closing but trapped” and “closing and likely to pass.”

---

### 1.4 Team/driver interaction features (MEDIUM-HIGH)

The best signal may not be raw driver form or raw team strength alone, but the interaction:

```text
attacker_driver_form × attacker_team_rank
defender_defend_rate × low-speed_track_type
team_drag_index × drs_zone_length
attacker_vs_defender_teammate_gap
constructor_matchup_history
```

The model should learn that some teams are more overtake-friendly on long straights while others defend better on traction-limited tracks.

## 2. Better Framing and Evaluation

### 2.1 Situation-aware pair model as primary output (HIGH)

The pair model is already the most useful practical framing. IP04 should make it the **main prediction surface**:

- “Will this battle result in an overtake in the next N laps?”
- “How likely is the attacker to pass before the next pit cycle?”

Suggested targets:

```text
pair_overtake_any
pair_overtake_within_2
pair_overtake_within_3
pair_overtake_before_battle_break
```

This aligns better with broadcast and strategy usage than next-lap only scoring.

---

### 2.2 Event-split validation (MEDIUM)

If team/driver form features are added, leakage risk rises.

**Proposal:**
- keep GroupKFold by race,
- add season-progressive evaluation,
- add “first 5 rounds vs rest of season” evaluation,
- add “new team ranking prior only” ablation for early-season robustness.

Also produce ablations:
- base v4,
- + team features,
- + driver form,
- + full situation features.

## 3. App Architecture Improvements

### 3.1 Split UI from inference engine (HIGH)

`models/app.py` currently mixes:
- model loading,
- feature engineering,
- business logic,
- HTML formatting,
- Gradio component wiring.

**Proposal:** split into layers:

```text
models/
  serving/
    loader.py          # model registry / metadata loading
    feature_builder.py # shared feature engineering for app + CLI
    predict.py         # pure inference API
    schemas.py         # typed request/response contracts
  app/
    gradio_ui.py       # Gradio only
```

Benefits:
- one source of truth for feature logic,
- easier testing,
- easier migration to FastAPI later,
- lower risk of notebook/app drift.

---

### 3.2 Model registry + schema versioning (HIGH)

Once v2/v3/v4/v5 coexist, the app should not rely on hard-coded filenames only.

**Proposal:**
- add a tiny model registry JSON:

```json
{
  "default": "v5",
  "models": {
    "v3": {"model": "...", "meta": "...", "data_version": "v3"},
    "v4": {"model": "...", "meta": "...", "data_version": "v4"}
  }
}
```

- validate input rows against expected features/data version,
- surface a friendly mismatch message when users upload incompatible CSVs.

---

### 3.3 FastAPI or API-first serving layer (MEDIUM)

Gradio is fine for demos, but a small API server would make the system reusable.

**Proposal:**
- expose `/predict/single`, `/predict/batch`, `/models/current`, `/health`,
- keep Gradio as a thin frontend client,
- let notebooks/CLI/browser all hit the same inference layer.

This also makes async batch scoring and future deployment much easier.

## 4. UI / UX Improvements

### 4.1 Explain the prediction, not just the number (HIGH)

The UI should answer:
- **Why** is the score high/low?
- **Which factors** matter most?
- **What changes** would alter the prediction?

**Proposal:** add a “Why this score?” panel:

```text
+ attacker closing rapidly
+ attacker on fresher tyres
+ stronger straight-line speed delta
- defender has track-position advantage
- low overtaking track type
```

This can be derived from SHAP values or simpler signed feature contributions.

---

### 4.2 Comparison / what-if mode (HIGH)

The best app feature for this use case is scenario comparison:

- same battle, different tyre offset,
- same battle, different gap,
- same battle, safety car restart vs green flag,
- same battle, different lap / stint age.

**Proposal:** add side-by-side compare cards:

```text
Scenario A | Scenario B | Delta P(overtake)
```

This is especially useful for race strategy analysis.

---

### 4.3 Better batch workflow (MEDIUM)

Current batch mode gives a table and CSV. For practical usage it should also provide:
- calibration / confidence summaries,
- race-level ranking,
- filters by team, driver, race, lap,
- downloadable “top opportunities” report,
- pair-level aggregation option.

---

### 4.4 Timeline / battle explorer (MEDIUM)

For a selected battle, show:
- lap-by-lap probability,
- actual outcome,
- gap trend,
- tyre age trend,
- DRS / flags markers.

This turns the app into an analysis tool instead of a single-score demo.

If P0–P1 from this proposal land cleanly:

| Metric | Current v4 | Target v5 (per-lap) | Target v5 (per-pair) |
|--------|------------|---------------------|----------------------|
| ROC-AUC (2025) | 0.894 | 0.90–0.91 | 0.90–0.92 |
| PR-AUC (2025) | 0.469 | 0.49–0.53 | 0.73–0.78 |
| Brier (2025) | 0.036 | 0.032–0.035 | 0.17–0.19 |
| Recall @ 30% precision | ~60% | 65–72% | 80%+ |

The biggest expected gain is not just raw metric lift, but **better situational correctness**: the model should know *who is attacking, in which car, in what season form, in what race context*.

That is the right direction for Formula 1 overtaking prediction.
