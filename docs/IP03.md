# Improvement Proposal 03: From Per-Lap Ranking to Actionable Battle Prediction

**Date:** 2026-03-19
**Status:** Proposed
**Precedes:** model_testing_4

---

## 0. Current State After IP02

The v3 model (XGBoost, Optuna-tuned, 56 features, trained on 2022–2024, v3 data with IP02 fixes) achieves:

| Metric | v2 (RF) | v3 (XGB) | Delta |
|--------|---------|----------|-------|
| ROC-AUC (CV) | 0.782 | 0.865 | +0.083 |
| PR-AUC (CV) | 0.149 | 0.384 | +0.235 |
| Brier (CV) | 0.029 | 0.122 | +0.093 |
| ROC-AUC (2025) | 0.785 | 0.888 | +0.103 |
| PR-AUC (2025) | 0.200 | 0.467 | +0.267 |
| Brier (2025) | 0.035 | 0.037 | +0.002 |

**Battle-pair level (2025 holdout):** ROC-AUC 0.910, PR-AUC 0.696.

Class separation improved from 0.07 to 0.40 (mean predicted probability for true overtakes: 0.651 vs 0.254 for non-overtakes).

### What worked

- Actual inter-car gap (LapStartTime) replaced the mislabelled pace-based gap — the single biggest data-quality fix.
- Pit-stop filtering removed 22% of "overtakes" that were pit-strategy shuffles.
- Gap-trend features (closing rate, gap delta) gave the model temporal signal it previously lacked.
- XGBoost with Optuna and `scale_pos_weight` dramatically improved class separation.

### What still needs work

1. **Brier score regressed** (0.035 → 0.037): the model ranks well but is poorly calibrated at the tails — it over-predicts on some non-overtakes and under-predicts on some true passes.
2. **Per-lap framing is fundamentally noisy**: an overtake happening on lap N means laps N−1 and N−2 of the same battle are labelled negative, even though the attack was building — the model is penalised for being directionally correct too early.
3. **No sequence modelling**: gap-trend features are a flat approximation of a dynamic process; they don't capture the full attack trajectory.
4. **Track-type coverage is unbalanced**: the model only saw two track types (`street` and `low-speed`) because `get_track_type()` maps most circuits to `street` as a fallback, masking real diversity.
5. **Feature space is large (56 features, 85+ after OHE)** with no pruning — likely includes several near-zero-importance features adding noise.

This proposal targets these gaps across four axes: problem reframing, modelling, feature refinement, and calibration.

## 1. Problem Reframing

### 1.1 Battle-pair aggregation as first-class model (HIGH)

IP02 §1.3A recommended restructuring the problem to predict per-battle-pair rather than per-lap. v3 evaluated at this level (ROC-AUC 0.910, PR-AUC 0.696) but still trained per-lap.

**Proposal:** Build a dedicated battle-pair model.

Each row represents one complete battle sequence (attacker A vs defender B for N consecutive laps). Features are aggregated:

```
min_gap, mean_gap, last_gap               # gap trajectory summary
gap_slope                                   # linear trend (closing rate)
max_pace_delta, mean_pace_delta             # pace advantage summary
n_laps, n_closing_laps                      # battle intensity
max_speed_delta_st                          # best straight-line advantage seen
tyre_age_at_start, tyre_age_at_end          # degradation window
compound_mismatch                           # boolean: different compounds
drs_available_fraction                      # fraction of battle laps in DRS zone
```

**Expected impact:** The training set becomes ~4,700 rows at ~16% positive rate (vs 11,300 rows at 6.7%). This is a far more balanced and statistically cleaner problem. The pair-level PR-AUC should improve from 0.696 to 0.75+.

---

### 1.2 Multi-horizon labels (MEDIUM)

Currently the label is binary: did the attacker gain position on the *next* lap? This is unnecessarily strict — many attacks build over 2–3 laps.

**Proposal:** Add two additional targets:
- `overtake_within_2`: position change within 2 laps
- `overtake_within_3`: position change within 3 laps

For the per-lap model, `overtake_within_3` is a softer, more learnable target that rewards the model for detecting the build-up phase. At inference, the model answers: "will an overtake happen soon?" rather than "will an overtake happen on exactly the next lap?"

## 2. Sequence Modelling

### 2.1 LSTM / GRU on battle windows (MEDIUM-HIGH)

The gap-trend features in v3 are hand-crafted summaries of a 3-lap window. A sequence model can learn richer temporal patterns from the raw feature trajectory.

**Architecture:**

```
Input: (batch, seq_len, n_features)  — e.g. (B, 5, 20) for 5-lap windows
  ↓
GRU(hidden=64, bidirectional=False)
  ↓
Dense(32, ReLU) → Dropout(0.3) → Dense(1, Sigmoid)
```

**Feature vector per timestep** (slim — ~20 features):
- gap_ahead, pace_delta, speed_st_delta, speed_fl_delta
- attacker_tyre_age, defender_tyre_age, tyre_age_difference
- race_progress, is_in_drs_zone, drs_zone_length
- attacker_position, defender_position
- safety_car, rainfall

**Training considerations:**
- Pad battles shorter than `seq_len` with zeros and use masking.
- Use the full battle sequence; the label applies to the final timestep only.
- GroupKFold by race to prevent leakage.

**Expected impact:** A GRU should capture the "catch → pressure → pass/fail" arc that hand-crafted features approximate. On battle-pair prediction, this could push PR-AUC above 0.75.

---

### 2.2 Hybrid: XGBoost + GRU stacking (LOW-MEDIUM)

The XGBoost model is strong on tabular features (track metadata, tyre compounds, weather). The GRU is strong on temporal dynamics. Stacking them:

```
meta_model = LogisticRegression()
meta_model.fit([xgb_proba, gru_proba], y)
```

This often adds 0.02–0.05 PR-AUC on imbalanced problems.

## 3. Feature Refinement

### 3.1 SHAP-based feature pruning (HIGH)

The v3 model uses 56 raw features expanding to 85+ after one-hot encoding. Many of these likely contribute near-zero information and add noise.

**Proposal:**
1. Compute SHAP values on the training set (TreeExplainer for XGBoost is fast).
2. Rank features by mean |SHAP|.
3. Remove the bottom 20% by importance.
4. Re-train and compare. Typically this maintains or slightly improves PR-AUC while reducing overfitting.

Also produce:
- SHAP summary plot (beeswarm) for the paper — shows which features drive overtake predictions.
- SHAP dependence plots for the top 5 features — reveals non-linear relationships the model learned.

---

### 3.2 Fix track_type coverage (HIGH)

The current `get_track_type()` in `track_info.py` only maps a handful of circuits explicitly; the rest fall through to `"street"`. This means the model sees almost no variation in the `track_type` feature.

**Current distribution (v3 train):**
- `street`: 10,468 rows (92%)
- `low-speed`: 876 rows (8%)

**Proposal:** Expand the mapping in `track_info.py` to cover all 2022–2025 circuits. At minimum:

```python
TRACK_TYPES = {
    "high-speed":   ["italy", "belgium", "great britain", "austria", "qatar", "saudi"],
    "medium-speed": ["spain", "japan", "abu dhabi", "bahrain", "china", "dutch", "emilia", "french"],
    "low-speed":    ["monaco", "hungary", "singapore"],
    "street":       ["azerbaijan", "miami", "las vegas", "australia"],
}
```

Regenerate v3 data after fixing the mapping.

---

### 3.3 Driver skill proxy without identity leakage (MEDIUM)

IP02 §2.6 proposed driver skill features. These were not implemented in v3. The safest approach:

```
attacker_overtake_rate = attacker's overtake attempts that succeeded / total attempts, rolling last 5 races
defender_defend_rate   = 1 − (overtakes conceded / battles faced), rolling last 5 races
```

Computed on historical data only (no future leakage). For the first 5 races of a season, fall back to the previous season's rate.

**Implementation:** Compute these as a post-processing step on the combined CSV before training, grouped by driver and ordered by race date.

---

### 3.4 Sector-level micro-features (MEDIUM)

FastF1 provides sector times (Sector1Time, Sector2Time, Sector3Time) per lap. The relative sector performance indicates where the attacker has an advantage:

```
sector1_delta = defender_S1_time − attacker_S1_time
sector2_delta = defender_S2_time − attacker_S2_time
sector3_delta = defender_S3_time − attacker_S3_time
strongest_sector = argmax(sector_deltas)  # which sector attacker is fastest in
```

An overtake is most likely to happen when the attacker's strongest sector leads into a DRS zone or heavy braking zone.

## 4. Calibration & Evaluation

### 4.1 Venn-Abers calibration (HIGH)

Isotonic calibration (used in v3) is unstable with small calibration sets and produces non-smooth probability curves. Venn-Abers predictors are a theoretically grounded alternative:
- Guaranteed valid coverage (unlike Platt/isotonic).
- Produces a probability interval [p_lower, p_upper] rather than a point estimate — useful for communicating uncertainty.

The `venn-abers` Python package implements this for sklearn-compatible classifiers.

**Expected impact:** Lower Brier score (fix the v3 regression) and more honest uncertainty communication.

---

### 4.2 Evaluation on unseen circuits (HIGH)

The current evaluation uses GroupKFold by race name, but every race name appears in every training year. The model has never been tested on a circuit it has never seen.

**Proposal:** Leave-one-circuit-out evaluation:
- For each circuit, train on all other circuits and predict on the held-out circuit.
- This tests generalization to new tracks (e.g., a brand-new street circuit added to the calendar).

---

### 4.3 Temporal-progressive evaluation (MEDIUM)

Train incrementally:
1. Train on 2022 only → evaluate on 2023
2. Train on 2022–2023 → evaluate on 2024
3. Train on 2022–2024 → evaluate on 2025

Plot metrics across these windows to detect concept drift (regulation changes, new cars, tyre spec changes). If performance drops, consider adding a season-weighting scheme (more recent seasons get higher weight).

---

### 4.4 Race-level case studies (MEDIUM)

Pick 2–3 races from the 2025 holdout and produce detailed prediction timelines:
- For each battle, show the predicted probability over time alongside the actual outcome.
- Overlay with race events (pit stops, safety cars, DRS activation).
- This is invaluable for the final report and for identifying systematic failure modes.

## 5. Engineering & Infrastructure

### 5.1 Reproducible experiment tracking (MEDIUM)

The current workflow saves one model artefact at a time with no experiment lineage. As the number of model variants grows, this becomes unmanageable.

**Proposal:** Integrate MLflow (lightweight, local):
- Auto-log Optuna trials, metrics, and parameters.
- Store artefacts with versioned run IDs.
- Compare experiments in the MLflow UI.

One-line setup: `mlflow.xgboost.autolog()` inside the Optuna objective.

---

### 5.2 Clear notebook outputs before committing (LOW)

Notebook outputs (plots, tables) inflate git diffs and repository size. Add a pre-commit hook or use `nbstripout` to clear cell outputs before committing.
