# IP06 — Dynamic model switching, probability UX, constructor standings, v6 roadmap

This document describes the IP06 implementation: **dynamic model switching** in the UI, diagnosis/mitigation of the **probability “ceiling”** issue, **constructor championship** data from `f1api.dev`, **supplemental schema fields** for richer `build_single_row` inputs, pipeline enrichment for training, **verdict label** cleanup, and a **v6 feature shortlist** with an evaluation framework.

---

## 1. Dynamic model switching

### Backend (existing)

- `GET /api/models/versions` — list versions from `registry.json`
- `POST /api/models/switch` — `{"version": "..."}` loads that artifact

### Frontend

- **`frontend/src/api/client.ts`** — `modelsVersions()`, `modelsSwitch(version)`
- **`frontend/src/components/model/ModelSwitcher.tsx`** — dropdown; if no versions are returned, shows the active model id as text
- **`frontend/src/pages/SingleBattle.tsx`** — loads versions, switches model, **refetches** `GET /api/models/schema`, clears prediction + sensitivity; **re-syncs circuit round/laps** after a switch (form reset)
- **`frontend/src/pages/ModelInfo.tsx`** — lists versions; click to switch and refresh current model info
- **`frontend/src/pages/BatchScoring.tsx`** — badge showing **active model version**

### Schema API extension

- **`GET /api/models/schema`** returns **`trained_feature_names`**: the feature list from the model artifact. The UI may append **supplemental** inputs (see §2) that are not trained columns; **1D sensitivity** only lists numeric features that appear in `trained_feature_names` so the backend does not reject unknown feature names.

## 2. Probability “ceiling” — diagnosis and fix

### Diagnosis

- Readonly / derived fields were stripped client-side before `POST /api/predict/single`, so the backend correctly used `build_single_row`.
- `build_single_row` applied **neutral defaults** for several high-impact fields (e.g. short `battle_duration`, weak `closing_rate` / gap trend, middling team rates). With **isotonic calibration** on a low base rate (~6.4% overtakes), those defaults **compressed** scores toward the middle.

### Mitigations (IP06)

1. **Expose impactful inputs in Basic mode** (`backend/app/services/feature_metadata.py`): e.g. `battle_duration`, `closing_rate`, team rates, `team_delta`, **`gap_delta_1`**, **`attacker_qualification_rank` / `defender_qualification_rank`**.
2. **`qualification_rank_difference`** is **readonly** and **derived** from the two qual ranks (`DERIVED_FROM`).
3. **Defaults** in `build_single_row` (`backend/app/services/feature_builder.py`): stronger gap trend default (**`-0.2`**), longer default **`battle_duration` (5 laps)**.
4. **Supplemental schema** (`SUPPLEMENTAL_FEATURES` in `backend/app/services/schema_builder.py`): UI fields that may not exist in older artifacts (e.g. v5 JSON) are still shown and sent to `build_single_row`; inference still uses only **`trained_feature_names`** for the XGBoost matrix.

### Constructor ranks (see §3)

- `attacker_constructor_rank`, `defender_constructor_rank`, `constructor_rank_delta` are filled in **`build_single_row`** when the request body is non-empty (live lookup). For **`build_single_row({})`** (schema defaults only), defaults **`10 / 10 / 0`** are used **without network I/O**.

## 3. Constructor championship (`f1api.dev`)

### Shared module

- **`pipeline/constructor_standings.py`** — HTTP fetch with **`lru_cache`**, `TEAM_ID_TO_APP_NAME` mapping, `fetch_constructors_standings`, `constructor_position_for_team`, `standings_positions_by_year_team`, `clear_standings_cache` (tests).
- **`backend/app/services/constructor_standings.py`** — re-exports from `pipeline` for the API layer.

### API

- **`GET /api/standings?year=YYYY`** — JSON for UI (`backend/app/routers/standings.py`).

### Inference row

- `build_single_row` adds constructor ranks and delta when **`raw` is non-empty** (see feature_builder).

### Frontend context

- **`frontend/src/components/battle/ConstructorContext.tsx`** — displays positions + Δ rank; notes that **v5 weights do not use** these columns until retraining.

### Training pipeline

- **`pipeline/team_features.py`** — `enrich_team_features()` appends **`attacker_constructor_rank`**, **`defender_constructor_rank`**, **`constructor_rank_delta`** using cached standings per season (snapshot from API for that year).

### v6 note

- **v5** artifact feature list does **not** include constructor columns. Adding them to **`models/artifacts/overtake_model_v6_meta.json`** (and retraining) is required for the model to consume them.

## 4. Verdict label bug (`_verdict_label`)

- **`backend/app/routers/predict.py`** — labels below the decision threshold use explicit **score bands** (e.g. “Below threshold · score band: low”) so bands do **not** overlap ambiguously (legacy duplicate **0.15** boundary between “Possible” / “Likely” style labels removed).

## 5. v6 feature shortlist & evaluation framework

Candidates are ranked by **expected mutual information (MI)** intuition; confirm with data before full integration.

### High expected MI

| Feature | Rationale |
|--------|------------|
| `constructor_rank_delta` | Explicit constructor strength vs implicit pace proxies |
| `race_phase` | Opening / mid / closing changes tyre, fuel, strategy |
| `defender_fresh_tyre` | Fresh rubber much harder to pass |
| `is_in_drs_zone` / `drs_zone_length` | Primary mechanical overtake aid |
| `safety_car` / `yellow_flag` | Restarts bunch the field |

### Medium expected MI

| Feature | Rationale |
|--------|------------|
| `attacker_stint` | Strategy interaction |
| `same_team` | Team-order / aggression effects |
| `rainfall` | Wet races change grip and opportunities |
| `sector` | Track-dependent overtake profiles |
| `pit_window_proximity` | Defender behaviour near stops |

### Interaction / derived (medium MI, tree splits)

- `tyre_compound_mismatch`, `fresh_tyre_advantage`, `drs_train_pressure`, etc.

### Evaluation steps (per candidate)

1. **Univariate MI** with `overtake` on v5 training data.
2. Add to pipeline, **retrain** with same hyperparameters → Δ **ROC-AUC** / **PR-AUC**.
3. **SHAP** importance + ablation.

## 6. v6 training plan (outline)

1. Regenerate training table with **constructor columns** (`enrich_team_features`) and any approved v6 features.
2. Update **`num_cols` / `cat_cols` / `features`** in model meta to match the notebook export.
3. Train **XGBoost + isotonic** (or chosen stack), export **`overtake_model_v6.pkl`** + **`overtake_model_v6_meta.json`**.
4. Register in **`registry.json`**, set **`DEFAULT_MODEL`** for deployment when ready.
5. Target metrics: document **PR-AUC** on hold-out seasons + calibration curve review.
